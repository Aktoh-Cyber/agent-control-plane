# ESLint Suppression Guidelines

**Version:** 1.0.0
**Last Updated:** 2025-12-08
**Status:** Official Project Standard

---

## Purpose

This document establishes clear guidelines for when ESLint suppressions are acceptable and how to properly document exceptions. Following these guidelines ensures code quality while providing flexibility when needed.

---

## Core Principles

### The Golden Rule

**Default Position:** Never suppress ESLint warnings or errors. Fix the underlying code instead.

**Exception Handling:** Only use suppressions when there is a legitimate technical reason AND the suppression is properly documented.

### Suppression Hierarchy (Preference Order)

1. **Fix the code** - Always the first choice
2. **Adjust the rule configuration** - If the rule doesn't fit the project
3. **Suppress for specific lines** - When fixing isn't possible
4. **Suppress for a file** - Rarely needed (usually generated code)
5. **Suppress globally** - Almost never appropriate

---

## When Suppressions Are Acceptable

### Category 1: Auto-Generated Code ✅ LEGITIMATE

**Always acceptable** for code generated by build tools.

**Examples:**

- WASM bindings generated by wasm-pack
- Native bindings generated by NAPI-RS
- GraphQL generated types
- Protocol buffer definitions
- Swagger/OpenAPI generated clients

**How to suppress:**

```typescript
/* eslint-disable */
/* prettier-ignore */
/* tslint:disable */

// auto-generated by [tool-name] - do not edit
```

**Location:** Top of file, with comment explaining the generator.

**Current usage in project:** 14 files (all legitimate)

---

### Category 2: Third-Party Type Issues ✅ ACCEPTABLE (With Documentation)

**Conditionally acceptable** when external library types are incomplete or incorrect.

**Example:**

```typescript
// eslint-disable-next-line @typescript-eslint/no-explicit-any
// Reason: @external/library v1.2.3 has incomplete type definitions
// TODO: Remove when https://github.com/external/library/issues/123 is resolved
const result: any = externalLibraryFunction();
```

**Requirements:**

1. Comment explaining the external library issue
2. Link to issue/PR if possible
3. TODO comment with resolution criteria
4. Specific line-level suppression (not file-level)

---

### Category 3: Intentional Rule Violations ✅ ACCEPTABLE (With Strong Justification)

**Rarely acceptable** when there's a strong technical reason to violate a rule.

**Example 1: Performance-critical code**

```typescript
// eslint-disable-next-line @typescript-eslint/no-unsafe-member-access
// Performance: Direct property access is 10x faster than type-safe alternative
// Benchmarked: see /benchmarks/property-access.bench.ts
const criticalValue = rawData.fastProperty;
```

**Example 2: Intentional any for generic utility**

```typescript
// eslint-disable-next-line @typescript-eslint/no-explicit-any
// Generic utility: Must accept any type for maximum flexibility
// Type safety is preserved through usage, not definition
export function deepClone<T>(obj: any): T {
  // ...
}
```

**Requirements:**

1. Clear explanation of why the rule violation is necessary
2. Evidence (benchmark, issue, design requirement)
3. Documented alternatives considered
4. Specific line-level suppression

## When Suppressions Are NOT Acceptable

### ❌ Developer Convenience

**Never acceptable** to suppress warnings just to avoid fixing code.

```typescript
// ❌ WRONG
// eslint-disable-next-line @typescript-eslint/no-unused-vars
const unused = getData(); // I might need this later
```

**✅ CORRECT:** Remove the unused variable or use it.

---

### ❌ Time Pressure

**Never acceptable** to suppress due to deadlines.

```typescript
// ❌ WRONG
// eslint-disable-next-line @typescript-eslint/no-explicit-any
// TODO: Fix before release
function quickFix(data: any) {
```

**✅ CORRECT:** Either fix it now or adjust your timeline.

---

### ❌ Misunderstanding the Rule

**Never acceptable** to suppress because you don't understand the rule.

```typescript
// ❌ WRONG
// eslint-disable-next-line prefer-const
// Not sure why this is flagged
let x = 5;
```

**✅ CORRECT:** Learn why the rule exists, then fix the code.

```typescript
const x = 5; // Use const for values that don't change
```

---

### ❌ Bulk Suppressions Without Review

**Never acceptable** to suppress entire categories without individual review.

```typescript
// ❌ WRONG
/* eslint-disable @typescript-eslint/no-explicit-any */
// Whole file uses any types - too much work to fix
```

**✅ CORRECT:** Fix types one by one, or re-architect the module.

## Suppression Syntax Guide

### 1. Specific Line Suppression (Preferred)

Use for single-line exceptions.

```typescript
// eslint-disable-next-line rule-name
const problematic = code();
```

Or inline:

```typescript
const problematic = code(); // eslint-disable-line rule-name
```

### 2. Multi-Line Suppression

Use for small blocks that must violate a rule.

```typescript
/* eslint-disable rule-name */
const line1 = problematic();
const line2 = code();
/* eslint-enable rule-name */
```

### 3. Multiple Rules

Specify multiple rules explicitly.

```typescript
/* eslint-disable rule-name-1, rule-name-2 */
const problematic = code();
/* eslint-enable rule-name-1, rule-name-2 */
```

### 4. File-Level Suppression (Rare)

Only for auto-generated files.

```typescript
/* eslint-disable */

// Auto-generated by tool-name v1.2.3
// DO NOT EDIT
```

## Documentation Requirements

### Minimum Documentation for Any Suppression

Every suppression (except auto-generated files) must have:

```typescript
// eslint-disable-next-line rule-name
// Reason: [Clear explanation of why suppression is needed]
// Context: [Link to issue, benchmark, or design doc if applicable]
const code = here();
```

### Template for Suppression Comments

```typescript
// eslint-disable-next-line rule-name
// Reason: [Why is the rule being violated?]
// Alternative considered: [What else was tried?]
// Impact: [What are the consequences?]
// Removal criteria: [When can this be removed?]
const code = here();
```

### Examples of Good Documentation

#### Good Example 1: Third-party library

```typescript
// eslint-disable-next-line @typescript-eslint/no-unsafe-call
// Reason: Legacy library 'old-sdk@1.0' has no TypeScript types
// Alternative: Considered writing .d.ts but library is deprecated
// Impact: Limited to this adapter module only
// Removal: When migrating to new-sdk@2.0 (planned Q2 2025)
const result = legacySDK.performAction();
```

#### Good Example 2: Performance optimization

```typescript
// eslint-disable-next-line @typescript-eslint/no-unsafe-member-access
// Reason: Direct property access for hot path (called 10M+ times/sec)
// Alternative: Type-safe accessor is 12x slower (see benchmarks/hot-path.ts)
// Impact: Carefully validated input ensures safety
// Removal: If V8 optimizes safe accessor to equal performance
const value = hotData.criticalProperty;
```

#### Good Example 3: Intentional design

```typescript
// eslint-disable-next-line @typescript-eslint/no-explicit-any
// Reason: Generic event system must accept any payload type
// Alternative: Union types don't scale to 100+ event types
// Impact: Type safety enforced at usage sites via generics
// Removal: Not planned - this is intentional design
export interface Event {
  type: string;
  payload: any; // Intentional - see EventEmitter<T> for typed usage
}
```

## Alternatives to Suppressions

### Alternative 1: Adjust ESLint Configuration

If a rule doesn't fit the project, adjust it in `config/eslint.config.js`.

```javascript
// Instead of suppressing no-console everywhere
rules: {
  'no-console': process.env.NODE_ENV === 'production' ? 'error' : 'warn',
}
```

### Alternative 2: Use More Specific Types

Instead of suppressing type warnings, improve types.

```typescript
// ❌ Suppression approach
// eslint-disable-next-line @typescript-eslint/no-explicit-any
function process(data: any) {

// ✅ Better types approach
function process<T extends Record<string, unknown>>(data: T) {
// or
function process(data: unknown) {
  if (isValidData(data)) {
    // Type narrowed here
  }
}
```

### Alternative 3: Refactor Code Structure

Sometimes the best fix is architectural.

```typescript
// ❌ Suppression approach
// eslint-disable-next-line complexity
function massiveFunction() {
  // 200 lines of complex logic
}

// ✅ Refactor approach
function wellFactoredFunction() {
  step1();
  step2();
  step3();
}
```

### Alternative 4: Create Type Guards

For unknown types, use type guards instead of any.

```typescript
// ❌ Suppression approach
// eslint-disable-next-line @typescript-eslint/no-explicit-any
function handle(data: any) {
  return data.property;
}

// ✅ Type guard approach
interface DataWithProperty {
  property: string;
}

function isDataWithProperty(data: unknown): data is DataWithProperty {
  return typeof data === 'object' && data !== null && 'property' in data;
}

function handle(data: unknown) {
  if (isDataWithProperty(data)) {
    return data.property; // Type-safe
  }
}
```

## Review Process

### Pre-Commit Checks

ESLint runs automatically via Husky pre-commit hooks. Suppressions will:

1. Be visible in git diffs
2. Trigger review comments if undocumented
3. Require justification in PR description

### Code Review Guidelines

**For Reviewers:**

When you see a suppression, check:

1. ✅ Is there a comment explaining why?
2. ✅ Is the reason legitimate (per this guide)?
3. ✅ Was an alternative approach considered?
4. ✅ Is the suppression as narrow as possible (line vs. file)?
5. ✅ Is there a plan to remove it eventually?

If any answer is "no", request changes.

**For Authors:**

Before requesting review:

1. Ensure every suppression has documentation
2. Use the suppression template above
3. Explain in PR description why suppressions were necessary
4. Link to relevant issues or design docs

## FAQs

### Q: What if I need to suppress multiple rules on one line?

**A:** List them explicitly, with a comment for each.

```typescript
/* eslint-disable-next-line
   @typescript-eslint/no-explicit-any,  // Reason for any
   @typescript-eslint/no-unsafe-call    // Reason for unsafe call
*/
const result = legacyLib.complexCall(data: any);
```

### Q: Can I suppress for the whole file if it's legacy code?

**A:** No. Fix issues incrementally or use granular suppressions. File-level suppressions hide future problems.

### Q: What if the lint rule is wrong?

**A:** Propose a change to the ESLint config. Rules should serve the team, not hinder it.

### Q: How do I know if my reason is "good enough"?

**A:** Ask: "Would this explanation make sense to someone reviewing my PR who doesn't know the context?" If no, expand it.

### Q: What about test files?

**A:** Test files have relaxed rules (see `config/eslint.config.js`). You rarely need suppressions. If you do, document them the same way.

### Q: Can I use eslint-disable-next-line without a specific rule?

**A:** No. Always specify which rule(s) you're disabling.

```typescript
// ❌ BAD
// eslint-disable-next-line
const code = here();

// ✅ GOOD
// eslint-disable-next-line @typescript-eslint/no-explicit-any
const code: any = here();
```

## Appendix: Common Rule Suppressions

### Rule: @typescript-eslint/no-explicit-any

**When acceptable:**

- Generic utilities requiring maximum flexibility
- Third-party library integration with no types

**When not acceptable:**

- Developer convenience
- Avoiding type definition work

**Better alternatives:**

- Use `unknown` and type guards
- Use generics: `<T>`
- Use utility types: `Record<string, unknown>`

---

### Rule: @typescript-eslint/no-unsafe-\*

**When acceptable:**

- Performance-critical paths (with benchmarks)
- Legacy library integration (with migration plan)

**When not acceptable:**

- New code
- When proper types are available

**Better alternatives:**

- Type assertions with validation
- Type guards and narrowing
- Wrapper functions with proper types

---

### Rule: no-console

**When acceptable:**

- CLI tools (intentional output)
- Development-only utilities

**When not acceptable:**

- Application code (use structured logging)
- Libraries (use event emitters)

**Better alternatives:**

- Winston logger (already implemented)
- Debug library for development
- Proper logging infrastructure

---

### Rule: @typescript-eslint/no-unused-vars

**When acceptable:**

- Function signatures matching interfaces
- Event handlers with unused parameters

**When not acceptable:**

- Dead code
- Incomplete implementations

**Better alternatives:**

- Prefix with underscore: `_unusedParam`
- Remove if truly unused
- Use TypeScript // @ts-expect-error for interface compliance
